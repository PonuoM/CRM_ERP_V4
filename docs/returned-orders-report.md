# คู่มือระบบหน้ารายงานออเดอร์ตีกลับ/ยกเลิก (Returned & Cancelled Orders Report)

**อัปเดตล่าสุด:** กรกฎาคม 2026
**Frontend:** `pages/ReturnedOrdersReportPage.tsx`
**Modal (จัดการรายละเอียด):** `components/ReturnedOrdersReport/OrderDetailsModal.tsx`
**Backend (API):** `api/Services/ReturnedOrdersReportService.php`

---

## 📌 วัตถุประสงค์ของระบบ
หน้านี้ถูกออกแบบมาเพื่อเป็นศูนย์กลางให้แอดมิน, QA, และทีมเซลส์ สามารถเข้ามาติดตาม วิเคราะห์ และจัดการออเดอร์ที่มีสถานะ **"ตีกลับ (Returned)"** หรือ **"ยกเลิก (Cancelled)"** โดยเฉพาะ ซึ่งมีฟีเจอร์เด่นคือการแนบไฟล์เสียงการโทร และการบันทึกสรุปผลการจัดการเคส (Resolution)

---

## 🛠️ โครงสร้างและการทำงาน (Features)

### 1. ระบบจัดการแท็บ (Tabs)
- แบ่งออกเป็น 2 แท็บหลักคือ:
  1. **ออเดอร์ตีกลับ (Returned):** กรองเฉพาะออเดอร์ที่ถูกส่งไปแล้วและตีกลับ
  2. **ออเดอร์ยกเลิก (Cancelled):** กรองเฉพาะออเดอร์ที่ถูกยกเลิกก่อนการส่ง

### 2. ระบบตัวกรองข้อมูล (Filters)
ส่งพารามิเตอร์ทั้งหมดไปประมวลผลที่ฝั่ง Backend (`ReturnedOrdersReportService.php`) 
- `resolution_status`: สถานะการจัดการ (All, Pending, Completed)
- `order_start_date` / `order_end_date`: กรองตาม **วันที่สั่งซื้อ**
- `action_start_date` / `action_end_date`: กรองตาม **วันที่ตีกลับหรือยกเลิก**
- `user_id`: กรองตามพนักงานขาย
- `audio_status`: กรองสถานะว่ามีไฟล์เสียงแนบหรือไม่ (has_audio, no_audio)
- `search_keyword`: ค้นหาอิสระ (ครอบคลุมรหัสออเดอร์, ชื่อลูกค้า, เบอร์โทร, ชื่อพนักงานขาย)

### 3. ระบบเรียงลำดับ (Client-Side Sorting)
- การคลิกที่หัวคอลัมน์ในตารางจะทำการเรียงลำดับข้อมูล **จากฝั่ง Frontend (Client-side)** ผ่าน `useMemo`
- **ข้อควรระวัง (Best Practice):** สำหรับคอลัมน์ที่เป็นตัวเงิน เช่น `total_amount` หรือ `returned_amount` ระบบจะทำการแปลงค่าเป็นตัวเลข (`parseFloat`) ก่อนเสมอ เพื่อป้องกันปัญหาการเรียงลำดับแบบตัวอักษร (String Sorting) ซึ่งจะทำให้ค่า 1000 มาก่อน 200

### 4. ลอจิกของ "ยอดทั้งออเดอร์" และ "ยอดตีกลับ"
เพื่อให้ข้อมูลแสดงผลถูกต้องเมื่อมีการตีกลับหรือยกเลิก:
- **ยอดทั้งออเดอร์ (total_amount):** ใช้ `orders.cod_amount` เป็นหลัก (ถ้าไม่มีให้ใช้ `orders.total_amount`) เพราะเมื่อออเดอร์ถูกตีกลับจนหมด ยอด `total_amount` เดิมอาจกลายเป็น 0
- **ยอดตีกลับ (returned_amount):**
  - กรณีตีกลับ: ใช้ผลรวม `SUM(cod_amount)` จากตาราง `order_boxes` เฉพาะกล่องที่สถานะเป็นการตีกลับ
  - กรณียกเลิก: ดึงค่ามาจาก `total_amount` ทั้งหมดโดยตรง

## 📊 ระบบการส่งออกข้อมูล (Export Data)

### 1. Export Data Grid (ตารางหลัก)
- **ประเภทไฟล์:** `.csv` และ `.xlsx`
- **วิธีทำงาน:**
  - นำข้อมูล `sortedData` ที่เห็นในตารางมา mapping คอลัมน์ใหม่
  - การจัดการข้อมูลซ้อนทับ (เช่น **รายการสินค้า** หรือ **ลิงก์ไฟล์เสียง** หลายรายการใน 1 ออเดอร์): ใช้เทคนิค **การขึ้นบรรทัดใหม่ในเซลล์เดียวกัน (`\n`)** แทนการสร้าง Row ซ้ำ เพื่อให้สามารถนำไฟล์ไปทำ Pivot Table ใน Excel ต่อได้โดยที่ข้อมูลระดับออเดอร์ไม่ผิดเพี้ยน

### 2. Export Case Summary (รายงานสรุปเคสรายบุคคล)
- อยู่ในส่วนของ Modal จัดการรายละเอียดออเดอร์ (`OrderDetailsModal.tsx`)
- **ประเภทไฟล์:** `.md` (Markdown) และ `.pdf` (Print to PDF)
- **วิธีทำงาน (.md):** ดึงข้อมูลลูกค้า, สินค้า, สรุปของแอดมิน, และประวัติไฟล์เสียง มาสร้างเป็นโครงสร้าง Markdown Text และใช้ Blob API ดาวน์โหลดออกมาทันที
- **วิธีทำงาน (.pdf):** 
  - ใช้ **Best Practice สำหรับภาษาไทย** โดยการสร้าง `<iframe>` ที่ซ่อนไว้ (Hidden) แล้วเขียนโครงสร้าง HTML/CSS จำลองกระดาษ A4 เข้าไป
  - จากนั้นใช้คำสั่ง `window.print()` เรียกหน้าต่างพิมพ์ของเบราว์เซอร์ เพื่อหลีกเลี่ยงปัญหาเรื่องสระลอยและฟอนต์เพี้ยนเมื่อเทียบกับการใช้ไลบรารีอย่าง jsPDF

## 📝 การจัดการข้อมูลใน Modal (OrderDetailsModal)
- การจัดการกับ **สถานะการพิมพ์/ข้อมูลที่ยังไม่บันทึก (isDirty):**
  - ไม่ใช้ `window.confirm` (Native popup) ตามเงื่อนไขของโปรเจกต์
  - ใช้การดักจับสถานะผ่านตัวแปร `showConfirmClose` และแสดง **กล่องข้อความแจ้งเตือนสีเหลือง** บริเวณปุ่มปิดแทน เพื่อความสวยงามและไม่รบกวน UX
- **Audio Links:** สามารถเพิ่ม ลบ หรือแก้ไขข้อมูลของลิงก์ไฟล์เสียง, วันที่โทร, และหมายเหตุได้ โดยระบบจะแยกลิงก์ที่มีอยู่แล้ว (`updated_audio_links` + `deleted_audio_ids`) และลิงก์ที่สร้างใหม่ (`new_audio_links`) ส่งไปอัปเดตที่ API ครั้งเดียว

## 📂 ฐานข้อมูลที่เกี่ยวข้อง
1. `orders`: ข้อมูลคำสั่งซื้อพื้นฐาน
2. `order_boxes`: ข้อมูลกล่องพัสดุ (ใช้คำนวณยอดตีกลับ)
3. `order_cancellations`: บันทึกข้อมูลและสาเหตุเมื่อออเดอร์ถูกยกเลิก
4. `order_audio_links`: บันทึกความสัมพันธ์ 1-to-many สำหรับลิงก์ไฟล์เสียงการโทรและออเดอร์
5. `order_audio_resolutions`: บันทึกข้อสรุปการทำงานของแอดมิน (Admin Notes) แยกจากตัวออเดอร์หลัก
