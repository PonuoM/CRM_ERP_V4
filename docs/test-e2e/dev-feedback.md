# ข้อเสนอแนะสำหรับทีม Dev เพื่อให้ทำ Automated E2E Test ได้ง่ายขึ้น

จากประสบการณ์ในการวางระบบและแก้ไขปัญหาโค้ด E2E Test (Playwright) สำหรับโปรเจกต์นี้ มี **ข้อเสนอแนะ (Feedback) เพื่อส่งต่อให้ทีม Dev** เพื่อให้การเขียนและดูแล Automated Test ในอนาคตมีความเสถียร (Stable) และจัดการได้ง่ายขึ้น ดังนี้ครับ:

## 1. 🎯 ใส่ `data-testid` ให้กับ Element สำคัญ
**ปัญหาที่เจอ:** ในโค้ดปัจจุบัน การทำ E2E ต้องพึ่งพา Selector ที่เปราะบางมาก เช่น การใช้ CSS Class ยาวๆ (เช่น `locator('.bg-white.p-4.rounded-lg...')`) หรือการหาด้วยข้อความ (เช่น `getByRole('button', { name: /นำเข้าข้อมูล/ })`) ซึ่งทำให้เกิด Error บ่อยๆ เมื่อมีเมนูชื่อซ้ำกัน หรือแค่เปลี่ยนสี/เปลี่ยนคลาส UI เทสต์ก็พังทันที

**สิ่งที่ควรปรับ:**
แนะนำให้ทีม Dev ใส่ attribute `data-testid="xxx"` ให้กับ **ปุ่ม, ช่องกรอกข้อมูล, Modal และ Tab ที่สำคัญๆ** เช่น:
- `data-testid="btn-save-order"`
- `data-testid="tab-pending-fetch"`
- `data-testid="input-cod-amount"`

การทำเช่นนี้จะช่วยให้ Playwright สามารถหา Element เจอได้แม่นยำ 100% ต่อให้มีการเปลี่ยนภาษา เปลี่ยนข้อความ หรือปรับดีไซน์ UI ก็ตาม

---

## 2. ⏳ จัดการสถานะ Loading (Spinner / Skeleton) ให้ชัดเจน
**ปัญหาที่เจอ:** ปัญหาหนักสุดที่ทำให้เทสต์ "พังแบบสุ่ม" (Flaky) คือ Race Condition จากจังหวะยิง API ตัวอย่างเช่น ตอนกดปุ่มเพิ่มสินค้าแล้วต้องรอรายชื่อสินค้าโหลดขึ้นมาบน Modal เทสต์มักจะกดข้ามไปเลยเพราะปุ่มเลือกสินค้ายังไม่โผล่ ส่งผลให้ทำรายการต่อโดยที่ตะกร้าว่างเปล่า

**สิ่งที่ควรปรับ:**
ควรมี Element ที่บ่งบอกสถานะกำลังโหลดที่ชัดเจนในระดับ DOM เช่น:
- `<div data-testid="loading-overlay">`
- `<div data-testid="table-skeleton">`

เพื่อให้คนเขียนเทสต์สามารถใช้คำสั่งง่ายๆ อย่าง `await page.getByTestId('loading-overlay').waitFor({ state: 'hidden' })` ได้ แทนที่จะต้องมานั่งกะเวลา `waitForTimeout(2000)` ซึ่งคาดเดาความเร็วอินเทอร์เน็ตไม่ได้

---

## 3. 📝 หลีกเลี่ยงการใช้ข้อความ (Label) ซ้ำกันในหน้าเดียว
**ปัญหาที่เจอ:** เกิดกรณี `Strict Mode Violation` ของ Playwright เพราะมีคำว่า **"นำเข้าข้อมูล"** ซ้ำกัน 2 ที่ (เมนูแถบข้างซ้าย กับ ปุ่มสีเขียวกลางหน้าจัดการออเดอร์) ทำให้เทสต์สับสนว่าต้องกดปุ่มไหน

**สิ่งที่ควรปรับ:**
ถ้านอกเหนือจากการใช้ `data-testid` (ตามข้อ 1) แล้ว ควรพยายามใช้คำศัพท์บนปุ่มให้ชัดเจน (Call to Action) หรือแยกประเภท Role ของ HTML ให้ถูกต้องตามหลัก Semantic เช่น:
- เมนูนำทาง ควรเป็น `<a href="... ">` หรือมี `role="link"`
- ปุ่มกระทำ (Action) ควรเป็น `<button>`

---

## 4. 🔗 การแสดงข้อมูลตัวเลขอ้างอิงที่ชัดเจน (Success State)
**ปัญหาที่เจอ:** หลังกดบันทึกคำสั่งซื้อสำเร็จ การจะดึง "เลขออเดอร์ (Order Ref)" เพื่อเอาไปให้ Worker ตัวอื่นใช้ค้นหาต่อในหน้า Backoffice ทำได้ยาก เพราะต้องใช้ Regex ไปดึงและตัด String เฉพาะคำว่า "หมายเลขคำสั่งซื้อ" ออกมา

**สิ่งที่ควรปรับ:**
ในหน้าต่าง Popup Success ควรมี Element เล็กๆ ที่ห่อหุ้มตัวเลขหรือรหัสที่ระบบ Generate ออกมาโดยเฉพาะ เช่น:
```html
<p>สร้างคำสั่งซื้อสำเร็จ: <span data-testid="generated-order-ref">260529-XXXXX</span></p>
```
การทำแบบนี้จะช่วยให้สคริปต์เทสต์ดึงข้อมูลเอาไปทำงานต่อแบบข้าม Role (เช่น ส่งจาก Telesale ไปให้ Backoffice ค้นหา) ได้ง่ายและเสถียรขึ้นมาก

---

## 5. 💡 ลดความซับซ้อนของ UI (เช่น การแบ่งยอด COD)
**ปัญหาที่เจอ:** การผูก State เปิด/ปิด ของปุ่ม "บันทึกคำสั่งซื้อ" ไว้กับการต้องกดปุ่ม "แบ่งยอดเท่าๆ กัน" ก่อน ค่อนข้างละเอียดอ่อนสำหรับบอท บางทีกดแล้ว State ของ React อัปเดตไม่ทัน ทำให้ตะกร้าแจ้งว่ายอด COD ไม่ถูกต้องและปุ่มบันทึกโดนล็อค

**สิ่งที่ควรปรับ (แง่ UX/Logic):**
สำหรับออเดอร์ทั่วไป (เช่น มีกล่องเดียว) ถ้าระบบสามารถ **Auto-fill ยอด COD ให้เท่ากับยอดสุทธิได้ทันที** ที่ผู้ใช้เลือกวิธีชำระเงินเป็น COD (โดยไม่ต้องให้ User หรือ Bot ต้องมากดปุ่มแบ่งยอดเอง) จะช่วยลดขั้นตอนการทำงาน (Friction) ได้ทั้งฝั่งคนใช้งานจริงและฝั่ง Automated Test ครับ

## 6. 📁 ไฟล์ที่ควรปรับปรุง (Target Files)
เพื่อให้สคริปต์ `tests/e2e/specs/core/core-flow.spec.ts` ทำงานได้เสถียรและดูแลรักษาง่ายขึ้น แนะนำให้ทีม Dev โฟกัสการเพิ่ม `data-testid` และปรับปรุง UI/UX ในไฟล์หลักๆ ดังต่อไปนี้:

1. **`pages/CreateOrderPage.tsx`**
   - **จุดที่ควรแก้:** ปุ่ม `+ เพิ่มสินค้า`, Modal เลือกสินค้า, ปุ่ม `แบ่งยอดเท่าๆ กัน`, และปุ่ม `บันทึกคำสั่งซื้อ`
   - **เหตุผล:** เป็นหน้าที่มี Race Condition เยอะที่สุด (ต้องรอโหลดข้อมูลสินค้า) และมีการจัดการ State ที่ซับซ้อน (โดยเฉพาะตอนเลือกชำระแบบ COD)

2. **`pages/ManageOrdersPage.tsx`**
   - **จุดที่ควรแก้:** Tab `รอดึงข้อมูล`, ฟอร์มค้นหาแบบ Advanced Filter, และปุ่ม `นำเข้าข้อมูล`
   - **เหตุผล:** เป็นจุดที่มีปัญหาปุ่มชื่อซ้ำซ้อนกับเมนู Sidebar (Strict Mode Violation) และการ Filter เลขออเดอร์ต้องเป๊ะมาก

3. **`pages/BulkTrackingPage.tsx`**
   - **จุดที่ควรแก้:** ตารางกรอก Tracking Number และปุ่มยืนยัน
   - **เหตุผล:** เพื่อให้บอทสามารถกรอกเลขพัสดุ (ที่ต้องเป็น Unique ID ไม่ซ้ำใน Database) ลงในช่อง Input ได้อย่างรวดเร็วและแม่นยำ

4. **ส่วนประกอบของ Modal / Popup (เช่น Success Modal)**
   - **จุดที่ควรแก้:** หน้าต่างที่เด้งขึ้นมาบอกว่าสร้างออเดอร์สำเร็จ
   - **เหตุผล:** ควรแยก `Order Ref` ออกมาหุ้มด้วยแท็กเฉพาะ (เช่น `<span data-testid="generated-order-ref">...</span>`) เพื่อให้เทสต์ดึงค่าไปใช้ต่อในหน้า Backoffice ได้โดยไม่ต้องพึ่งพา Regex ที่เสี่ยงพัง

5. **`components/OrderManagementModal.tsx` และโฟลว์ตรวจสอบสลิป**
   - **จุดที่ควรแก้:**
     - เพิ่ม `data-testid="tab-waiting-verify-slip"` ให้แท็บ "รอตรวจสอบสลิป" (ใน ManageOrdersPage)
     - เพิ่ม `data-testid={"btn-manage-order-${order.order_ref}"}` หรือ ID เฉพาะ ให้ปุ่ม "จัดการ" ในแต่ละแถวของตารางออเดอร์
     - เพิ่ม `data-testid="checkbox-verify-slip"` ให้กับ Checkbox ติ๊กยืนยันสลิปใน Modal
     - เพิ่ม `data-testid="btn-confirm-slip"` ให้กับปุ่ม "ยืนยันสลิป / ยืนยันเรียบร้อย" ใน Modal
   - **เหตุผล:** สำหรับออเดอร์ประเภท **โอนเงิน (Transfer)** บอทจำเป็นต้องกดเข้าไปยืนยันสลิปในแท็บรอดำเนินการก่อน จึงจะสามารถข้ามไปหน้าส่งออก (Export) ได้ การมี testid จะช่วยให้บอทเข้าไปกด Checkbox ได้อย่างแม่นยำ

## 7. ⚠️ การค้นหา Element ด้วย `getByText({ exact: true })` ใน Autocomplete / Dropdown
**ปัญหาที่เจอ:** ในส่วนของการเลือกที่อยู่ (เช่น ตำบล/แขวง) หากใช้คำสั่ง `getByText('ปากน้ำ', { exact: true })` จะหาไม่เจอ เนื่องจากใน UI ปัจจุบันมีการเรนเดอร์ข้อความเสริมต่อท้าย เช่น รหัสไปรษณีย์ `ปากน้ำ (10270)` ทำให้ `exact: true` ทำงานผิดพลาด

**สิ่งที่ควรปรับ:**
แนะนำให้ทีม Dev พิจารณาเพิ่ม `data-testid` หรือระบุ attribute เฉพาะลงในตัวเลือก (Options) ของ Autocomplete / Dropdown ทุกตัว เช่น:
```jsx
<div data-testid={`option-subdistrict-${subDistrict.id}`}>
  {subDistrict.name_th} ({subDistrict.zip_code})
</div>
```
เพื่อให้ฝั่ง Automated Test สามารถเลือก Item ได้อย่างแม่นยำ โดยไม่ต้องใช้ `getByText` แบบ `{ exact: false }` ที่อาจเกิดความเสี่ยงในการจับเจอชื่อซ้ำกันได้ครับ

---
