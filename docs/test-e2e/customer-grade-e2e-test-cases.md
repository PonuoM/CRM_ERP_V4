# 🧪 E2E Test Cases สำหรับ Customer Grade System

ไฟล์นี้สรุปฟังก์ชันและคู่มือสำหรับ QA/Tester ในการเขียนและทำการทดสอบระบบ Dynamic Customer Grade แบบ End-to-End (E2E) ด้วย **Playwright** หรือเครื่องมือ Automation Testing อื่นๆ

---

## 🎯 1. ฟีเจอร์ที่ต้องทำการทดสอบ (Features to Test)

### 1.1 หน้าจัดการตั้งค่าเกรดลูกค้า (Customer Grade Manager UI)
1. **[UI-01] การเพิ่มเกรดใหม่ (Create Grade)**
   - **Action:** กรอกชื่อเกรด (เช่น `VIP`), ใส่ยอดขั้นต่ำ (เช่น `100000`), เลือกสี, แล้วกดบันทึก
   - **Expect:** ข้อมูลแสดงผลบนหน้าจอ และแจ้งเตือน "บันทึกข้อมูลเรียบร้อย" (สีเขียว)
2. **[UI-02] ตรวจสอบ Validation ยอดสะสมขั้นต่ำห้ามติดลบ**
   - **Action:** ใส่ยอดขั้นต่ำติดลบ (เช่น `-5000`) แล้วกดบันทึก
   - **Expect:** เด้ง Toast/Error เตือน "ยอดสะสมขั้นต่ำไม่สามารถติดลบได้" และไม่มีการบันทึกลง Database
3. **[UI-03] ตรวจสอบ Validation ชื่อเกรดห้ามว่างหรือซ้ำกัน**
   - **Action:** เพิ่มชื่อเกรดซ้ำกัน 2 แถว (เช่น `A` กับ `A`) แล้วกดบันทึก
   - **Expect:** เด้ง Error เตือน "มีชื่อเกรดซ้ำกัน กรุณาแก้ไข"
4. **[UI-04] การจัดเรียงลำดับ (Auto-Sort on Save)**
   - **Action:** เพิ่มเกรดยอดเงิน `5000` ไว้ด้านบน และ `150000` ไว้ด้านล่าง จากนั้นกดบันทึก
   - **Expect:** เมื่อบันทึกเสร็จ หน้าจอต้องสลับแถวให้ `150000` ขึ้นไปอยู่ด้านบน (เรียงจากมากไปน้อย)
5. **[UI-05] การลบเกรด (Delete Grade)**
   - **Action:** กดไอคอนถังขยะในแถวของเกรด แล้วกดบันทึก
   - **Expect:** เกรดถูกลบออกจากหน้าจออย่างสมบูรณ์

### 1.2 การประมวลผลคำนวณยอดแบบกลุ่ม (Batch Recalculation)
1. **[BATCH-01] หน้าต่าง Confirm Modal**
   - **Action:** คลิกปุ่ม "ประมวลผลเกรดลูกค้าทั้งหมด"
   - **Expect:** จะต้องมีหน้าต่าง Confirm Modal (ไม่ใช่ `window.confirm` ธรรมดา) เด้งขึ้นมาให้ยืนยันการประมวลผล
2. **[BATCH-02] อัปเดตข้อมูลลูกค้า (Recalculate Grades)**
   - **Action:** คลิกปุ่ม "เริ่มประมวลผล" บน Confirm Modal
   - **Expect:** ระบบจะโชว์ Loading เมื่อเสร็จสิ้นจะแสดงแจ้งเตือนว่าอัปเดตเกรดเสร็จสิ้นแล้ว (สามารถเช็คด้วยการเข้าไปดูหน้ารายชื่อลูกค้าว่าเกรดถูกอัปเดตตรงตามยอดซื้อหรือไม่)

### 1.3 การคำนวณแบบจำกัดช่วงเวลา (Time Range & Calc Modes)
*(ต้องใช้ API หรือ Playwright สั่งสร้าง Order ทดสอบแบบเจาะจงวันที่)*
1. **[CALC-01] โหมด "คิดทุกออเดอร์" (All)**
   - **Action:** ตั้งค่าเป็น `all` -> ลูกค้ามียอด 10,000 (ปีที่แล้ว) และ 20,000 (วันนี้)
   - **Expect:** เมื่อประมวลผล เกรดจะคำนวณจากยอด 30,000 บาท
2. **[CALC-02] โหมด "คิดตามช่วงเวลาคงที่" (Fixed Date)**
   - **Action:** ตั้งเวลา Fixed: `2026-01-01` ถึง `2026-12-31` -> ลูกค้ามียอดสั่งซื้อ 10,000 ในปี 2025 และ 20,000 ในปี 2026
   - **Expect:** ระบบจะคำนวณจากยอด 20,000 เท่านั้น
3. **[CALC-03] โหมด "นับย้อนหลัง" (Relative Days)**
   - **Action:** ตั้งค่านับย้อนหลัง `30` วัน -> มีออเดอร์อายุ 10 วัน (10,000 บาท) และออเดอร์อายุ 40 วัน (20,000 บาท)
   - **Expect:** ระบบจะคำนวณจากยอด 10,000 บาทเท่านั้น

### 1.4 การอัปเดตแบบ Real-time (Real-time Grade Calculation)
1. **[RT-01] การเลื่อนขั้นเกรดเมื่อสถานะออเดอร์เปลี่ยน**
   - **Action:** สร้างออเดอร์ใหม่มูลค่าที่ทำให้ยอดรวมถึงเกณฑ์ `VIP` -> ปรับสถานะเป็น "จัดส่งสำเร็จ (Delivered)" (ถ้าใช้โหมด `delivery_date`)
   - **Expect:** เข้าไปดูหน้ารายละเอียดลูกค้า (Customer Detail) เกรดของลูกค้าคนนั้นจะเปลี่ยนเป็น `VIP` ทันที โดยไม่ต้องกดปุ่มประมวลผลแบบกลุ่ม

### 1.5 การแยกข้อมูลระหว่างบริษัท (Multi-tenant Isolation)
1. **[MULTI-01] บริษัทที่ 1 กับบริษัทที่ 2 ต้องมีเกรดคนละชุด**
   - **Action:** ล็อกอินด้วยแอดมิน บริษัท A แก้ไขชื่อเกรดเป็น `A-Grade` -> จากนั้นล็อกอินด้วยบริษัท B
   - **Expect:** บริษัท B จะยังคงเห็นเกรดปกติของตนเอง และลูกค้าระหว่าง 2 บริษัทจะคำนวณยอดไม่ก้าวก่ายกัน

---

## 🛠 คำแนะนำในการเขียน Playwright Script

1. **เตรียม Test Data:** 
   - ก่อนทำ `[CALC-01]` ถึง `[CALC-03]` แนะนำให้ยิง API เข้าไปสร้างฐานข้อมูล `orders` ให้มี Timestamp อดีต ปัจจุบัน อนาคต เตรียมไว้สำหรับ User Test
   - หลีกเลี่ยงการพึ่งพาข้อมูลจริง (Production Data) ในการ Test 
2. **เช็ค Selectors:**
   - ใช้ `data-testid` หรือหา Text ควบคู่เวลาหาสีของเกรด เช่น `expect(page.locator('.badge')).toHaveText('VIP')`
3. **การทำงานของ Modal:**
   - การคลิกปุ่มประมวลผล จะทำให้ปรากฏ `div[role="dialog"]` ขึ้นมาบน DOM (ให้เช็ค Dialog แบบ Custom Component ไม่ใช่ `page.on('dialog')` ของเบราว์เซอร์ปกติ)
