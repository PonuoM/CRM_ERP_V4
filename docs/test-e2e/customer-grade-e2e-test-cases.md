# 🧪 E2E Test Cases: Customer Grade System (Isolated DB)

**Testing Strategy:** E2E Testing on Isolated Test Database  
**Target Environment:** Local Database `primacom_mini_erp_test`  

เอกสารนี้รวบรวม Test Cases สำหรับระบบ Dynamic Customer Grade โดยอ้างอิงหลักการ E2E Best Practice (Data Seeding -> Given/When/Then -> Teardown) เพื่อให้มั่นใจว่าการทดสอบไม่กระทบข้อมูล Production และจำลองเหตุการณ์ได้สมจริงที่สุด

---

## 🏗 Pre-requisites (การเตรียม Environment)
1. **Database Isolation:** ระบบ Automation ต้องเชื่อมต่อกับฐานข้อมูล `primacom_mini_erp_test` เท่านั้น
2. **Data Seeding:** ก่อนเริ่มรันแต่ละ Test Suite จะต้องมีการล้างข้อมูลเดิม (Teardown) และ Insert ข้อมูลลูกค้า, บริษัท, และออเดอร์จำลอง (Fixtures) ใหม่เสมอ
3. **API Access:** Playwright/Cypress จะต้องยิง API ไปยัง Endpoint ของ Test Environment

---

## 🎯 Test Cases (BDD Format: Given / When / Then)

### 1. หมวดหมู่ UI & Validation (หน้าจัดการเกรดลูกค้า)

**[TC-UI-01] การเพิ่มเกรดใหม่ (Create Grade)**
- **Given** ผู้ดูแลระบบ (Admin) อยู่ในหน้า "จัดการระดับลูกค้า"
- **When** ผู้ดูแลระบบเพิ่มเกรดชื่อ `VIP`, กำหนดยอดขั้นต่ำ `100000`, เลือกสีเหลืองทอง และกด "บันทึก"
- **Then** ระบบต้องแสดงข้อความ "บันทึกข้อมูลเรียบร้อย" 
- **And** ต้องมีข้อมูลเกรด `VIP` ถูกบันทึกลงตาราง `customer_grades_config` ใน `primacom_mini_erp_test`

**[TC-UI-02] การจัดเรียงลำดับเกรดอัตโนมัติ (Auto-Sort on Save)**
- **Given** ผู้ดูแลระบบอยู่ในหน้า "จัดการระดับลูกค้า"
- **When** ผู้ดูแลระบบเพิ่มเกรดขั้นต่ำ `5,000` ไว้ด้านบนสุด และ `150,000` ไว้ด้านล่างสุด จากนั้นกด "บันทึก"
- **Then** หน้าจอ UI ต้องจัดเรียงแถวใหม่ โดยดึง `150,000` ขึ้นไปไว้บรรทัดบนสุด (เรียงจากยอดมากไปน้อย)

### 2. หมวดหมู่ การคำนวณแบบจำกัดช่วงเวลา (Time Range Logic)

**[TC-CALC-01] โหมดคิดตามช่วงเวลาคงที่ (Fixed Date)**
- **Given** มีข้อมูลลูกค้า A ในตาราง `customers` 
- **And** ลูกค้า A มีออเดอร์ที่จัดส่งสำเร็จในปี 2025 ยอด `10,000` และในปี 2026 ยอด `20,000` อยู่ในตาราง `orders`
- **When** แอดมินตั้งค่าโหมดคำนวณเป็น `Fixed Date` (1 ม.ค. 2026 - 31 ธ.ค. 2026) และกด "ประมวลผลเกรดลูกค้าทั้งหมด"
- **Then** เกรดของลูกค้า A ในตาราง `customers` ต้องถูกคำนวณจากยอด `20,000` บาทเท่านั้น

**[TC-CALC-02] โหมดนับย้อนหลัง (Relative Days)**
- **Given** มีข้อมูลลูกค้า B ในระบบ
- **And** ลูกค้า B มีออเดอร์เมื่อ 10 วันก่อน ยอด `10,000` และเมื่อ 40 วันก่อน ยอด `20,000`
- **When** แอดมินตั้งค่าโหมดคำนวณเป็น `Relative` ย้อนหลัง `30` วัน แล้วสั่งประมวลผล
- **Then** เกรดของลูกค้า B ต้องถูกคำนวณจากยอด `10,000` บาทเท่านั้น (ตัดออเดอร์ 40 วันก่อนทิ้ง)

### 3. หมวดหมู่ การอัปเดตแบบ Real-time (Event-Driven)

**[TC-RT-01] อัปเดตเกรดทันทีเมื่อปิดการขายสำเร็จ**
- **Given** ลูกค้า C มีเกรดปัจจุบันเป็น `Normal`
- **And** แอดมินตั้งเกณฑ์เกรด `Super VIP` ไว้ที่ยอด `150,000` บาท
- **When** พนักงานขายสร้างออเดอร์ให้ลูกค้า C มูลค่า `150,000` บาท และอัปเดตสถานะเป็น `Delivered`
- **Then** ระบบต้อง Trigger การคำนวณเกรดใหม่ทันที
- **And** เมื่อดึงข้อมูลลูกค้า C จากฐานข้อมูล เกรดต้องเปลี่ยนเป็น `Super VIP` อัตโนมัติโดยไม่ต้องรอ Cron Job

### 4. หมวดหมู่ Multi-tenant Isolation (การแยกข้อมูลบริษัท)

**[TC-MULTI-01] การตั้งค่าเกรดต้องไม่กระทบข้ามบริษัท**
- **Given** มีฐานข้อมูลบริษัท A (`company_id=1`) และ บริษัท B (`company_id=2`)
- **When** แอดมินบริษัท A แก้ไขชื่อเกรดสูงสุดเป็น `A-Grade` และกดบันทึก
- **Then** การเปลี่ยนแปลงต้องเกิดขึ้นกับ `company_id=1` เท่านั้น
- **And** เมื่อแอดมินบริษัท B ล็อกอินเข้ามา ต้องยังคงเห็นเกรดดั้งเดิม (D, C, B, A) ของตัวเอง

### 5. หมวดหมู่ Cron Job (ระบบกวาดล้างและอัปเดตเกรดอัตโนมัติ)

**[TC-CRON-01] การอัปเดตเกรดลูกค้าทั้งหมดผ่าน Cron Script**
- **Given** มีลูกค้า D ที่เคยได้เกรด `VIP` จากยอดขายเมื่อปีที่แล้ว
- **And** แอดมินได้ตั้งค่าโหมดคำนวณเป็น "นับย้อนหลัง 30 วัน" (ทำให้ลูกค้า D ควรจะตกเกรดเพราะไม่มีออเดอร์ใหม่)
- **When** รันคำสั่ง CLI `php api/cron/recalculate_all_grades.php`
- **Then** สคริปต์ต้องทำงานสำเร็จและแสดงผลการประมวลผล (เช่น `Successfully updated X customers.`)
- **And** เมื่อ Query ฐานข้อมูลลูกค้า D ใน `primacom_mini_erp_test` เกรดจะต้องถูกปรับลดลงจาก `VIP` เป็นเกรดที่ถูกต้องตามยอดขาย 30 วันย้อนหลัง (เช่น `D`) ทันที
