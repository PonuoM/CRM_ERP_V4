# 📦 Distribution V2 (Smart Allocation Architecture)

ระบบแจกงาน Distribution V2 คือ **"ศูนย์กลางการกระจายลูกค้าระดับสูง"** ที่ใช้สำหรับนำรายชื่อลูกค้าจากถังต่างๆ (Baskets) มาแจกจ่ายให้กับพนักงาน Telesale แต่ละคน โดยมาพร้อมกับ **สถาปัตยกรรม Component-Based แบบใหม่ล่าสุด** และ **ระบบสมองกลป้องกันรายชื่อซ้ำซ้อน (Smart Allocation)**

---

## 🏗️ 1. สถาปัตยกรรมระบบ (System Architecture)

เพื่อลดปัญหา Code Monolith (ไฟล์เดียว 3,500 บรรทัด) ระบบ Distribution ปัจจุบันถูกเขียนด้วยหลักการ **Separation of Concerns** โดยแบ่งเป็นส่วนๆ ดังนี้:

### 📂 ไฟล์หลัก (The Orchestrator)
- **`pages/CustomerDistributionV2.tsx`**: ทำหน้าที่เป็นคอนเทนเนอร์หลัก (Container) เก็บเฉพาะ State ที่ใช้ร่วมกันระดับบนสุด และควบคุมการทำงานภาพรวม (เหลือเพียง ~700 บรรทัด)

### 🧩 Component ย่อย (The Modules)
แยกการทำงานตามหน้าต่าง (UI Sections) ไปไว้ที่ `components/DistributionV2/`:
1. **`DistributionFilterForm.tsx`**: จัดการเรื่องการค้นหาและฟิลเตอร์ลูกค้า (วันเวลา, แท็ก, สินค้า, ฯลฯ)
2. **`DistributionTelesaleTable.tsx`**: จัดการตารางรายชื่อพนักงาน, การเช็คเวลาโทร, ระบบเลือกตัวพนักงานและกรองด้วย Supervisor
3. **`DistributionTargetSection.tsx`**: จัดการหน้าต่างตั้งค่าการแจกงาน (เป้าหมาย, เลือกโหมด Equal/Load Balance/Performance)
4. **`DistributionPreviewModal.tsx`**: แสดง Modal ตัวอย่างลูกค้าที่จะถูกแจกและคำเตือนเรื่องรายชื่อซ้ำซ้อน

---

## 🧠 2. กลไกแจกงานอัจฉริยะ (Smart Allocation Engine)

จากเดิมที่ระบบ V2 เก่าใช้เพียงคณิตศาสตร์หารเฉลี่ยตายตัว (ซึ่งทำให้เกิดปัญหารายชื่อแจกซ้ำพนักงานคนเดิม) ระบบปัจจุบันถูกอัปเกรดให้ **"คิดก่อนแจก"**:

### ✅ ขั้นตอนที่ระบบประมวลผลอัตโนมัติ:
1. **ดึงประวัติความขัดแย้ง (Conflict Checking):** ทันทีที่กดแจก ระบบจะเรียก API `get_assign_checks` เพื่อกวาดดูว่าลูกค้าในล็อทนี้ **"ใครเคยดูแลมาแล้วบ้าง"**
2. **Prioritize ลูกค้ายากสุดก่อน (Most Constrained First):** ลูกค้าคนไหนที่ผ่านมือพนักงานมาหลายคนแล้ว จะถูกนำมาจัดสรรก่อนคนอื่น เพราะหาที่ลงยากที่สุด
3. **หลบหลีกรายชื่อซ้ำ (Duplicate Evasion):** ระบบจะแจกลูกค้าให้กับ **"พนักงานที่ไม่เคยได้ลูกค้ารายนี้มาก่อน"** เท่านั้น
4. **เฉลี่ยปริมาณงาน (Load Balancing):** ในกลุ่มพนักงานที่ไม่ซ้ำ ระบบจะเลือกให้คนที่ถือยอดในรอบนั้นน้อยที่สุดก่อน เพื่อรักษาสมดุลแบบ Equal
5. **แจ้งเตือนก่อนพลาด (Fallback Warning):** หากมีลูกค้าที่ **"ซ้ำกับพนักงานทุกคนที่เลือกมา"** (ไม่มีทางเลี่ยงได้) ระบบจะขึ้นคำเตือนชัดเจนในหน้า Preview ให้หัวหน้าตัดสินใจ

---

## ⚙️ 3. โหมดการแจกงาน (Distribution Modes)

1. **แบ่งตามประสิทธิภาพ (Performance):** แจกให้ตาม % นาทีการโทร (Call Minutes) ใครโทรเยอะได้เยอะ
2. **แบ่งเท่ากันอัจฉริยะ (Smart Equal):** หารจำนวนแจกเท่าๆ กันทุกคน แต่จะแทรกแซงด้วย Smart Allocation เพื่อไม่ให้แจกรายชื่อที่เคยซ้ำ
3. **ปรับสมดุล (Load Balance):** มองจำนวนลูกค้าทั้งหมดที่ถืออยู่ ใครถือน้อยสุดจะได้เติมให้เต็มก่อน
4. **กำหนดเอง (Manual Quota):** กรอกตัวเลขเป้าหมายด้วยมือเป็นรายบุคคล

---

## ⚡ 4. การบริหารจัดการรายชื่อแบบกลุ่ม (Bulk Actions)

สำหรับหัวหน้าทีมที่ต้องการย้ายหรือดึงคืนลูกค้าจำนวนมากๆ ในคลิกเดียว:

- **⚡ ดึงคืน/โอน (ทั้งหมด):** ดำเนินการย้ายรายชื่อของพนักงานทุกคนในบริษัทพร้อมกันเข้าถังกลาง
- **ระบบคัดกรองพฤติกรรม:** เลือกดึงงานได้ตามพฤติกรรม เช่น:
  - **ไม่มีการโทร:** ไม่รับสาย ไม่เคยโทร
  - **โทรแล้วไม่นัด:** โทรคุยแล้วแต่ปิดการขายหรือนัดหมายไม่ได้
  - **โทรและนัด:** ลูกค้าที่มี Potential มีประวัติการนัด (ซึ่งปกติไม่ควรดึงคืน)
- **Smart Cap (กันดึงเกิน):** พิมพ์ลิมิตจำนวนคนที่จะดึงได้ ระบบจะตรวจจับและดึงให้สูงสุดเท่าที่มีจริง ไม่เกิด Error

---

## 🔄 5. การรีเซ็ตรอบ (Manual Reset)

เมื่อต้องการวนลูกค้ารอบใหม่ให้พนักงานคนเดิม (เริ่ม Life Cycle ใหม่):
1. เข้าเมนู **"Manual Reset"** (มุมขวาบน)
2. เลือกรอบที่ต้องการ หรือค้นหาตามเบอร์โทร/รายชื่อ Agent
3. กดยืนยันเพื่อล้างตาราง `customer_assign_check` สำหรับลูกค้ารายนั้นๆ
4. ผลลัพธ์: พนักงานคนเดิมจะได้รับลูกค้ารายนี้อีกครั้งโดยไม่ติดตัวกรอง Smart Allocation

---

## 🚫 6. การปลดบล็อคลูกค้า (Unblock Target)

ลูกค้าในตะกร้า **"ลูกค้าบล็อค"** ไม่สามารถกดแจกตรงๆ ได้ ต้องผ่านกระบวนการปลดบล็อค:
1. กดการ์ด 🚫 ลูกค้าบล็อค
2. เลือกรายชื่อที่ต้องการให้โอกาส
3. เลือกปลายทาง (เช่น ถัง 6-12 เดือน)
4. กด **"✅ ปลดบล็อค"** → สถานะบล็อคจะถูกล้างและกลับเข้าสู่ Flow แจกงานปกติ

---

## 📊 7. ระบบประวัติและรายงานการแจกงาน (Distribution Report & Export)

หลังจากการแจกงานเสร็จสิ้น ระบบจะทำการ **Snapshot** ข้อมูลการแจกงานรอบนั้นๆ เอาไว้โดยอัตโนมัติ เพื่อให้หัวหน้างานสามารถตรวจสอบย้อนหลังและนำออกข้อมูลได้:

### 📂 การทำงานของ Database (Session Logging)
- **`distribution_sessions`**: เก็บข้อมูลภาพรวมของการแจก 1 ครั้ง (ใครเป็นคนแจก, โหมดอะไร, แจกไปกี่คน) และมีการเก็บข้อมูลเพิ่มเติมดังนี้:
  - **`agent_snapshot`**: บันทึกสถานะของพนักงานทุกคน ณ วินาทีที่แจก (เวลาโทรปัจจุบัน, จำนวนที่เข้าเกณฑ์/หลุดเกณฑ์)
  - **`source_basket`**: บันทึกชื่อตะกร้าตั้งต้นที่ทำการดึงรายชื่อมาแจก (เช่น `mid_6_9m`) เพื่อให้ทราบแหล่งที่มา
- **`distribution_session_details`**: เก็บความสัมพันธ์แบบ 1-to-Many ว่าพนักงานคนไหน ได้รับลูกค้าเบอร์อะไรบ้าง

### 🖥️ การแสดงผลหน้าจอประวัติ (Report UI Enhancements)
เพื่อการตรวจสอบที่ง่ายและเป็นระเบียบ หน้าจอแสดงประวัติได้ถูกออกแบบมาดังนี้:
1. **ระบบย่อ/ขยาย (Expand & Collapse):** แสดงผลเป็น Accordion หากต้องการดูรายชื่อพนักงานที่ได้รับงาน สามารถกดปุ่มลูกศร (Chevron) เพื่อกางหรือพับเก็บได้ ช่วยประหยัดพื้นที่เมื่อมีรอบแจกงานจำนวนมาก
2. **รูปแบบตาราง (Table Layout):** รายชื่อพนักงานและจำนวนที่ได้รับ จะถูกจัดเรียงเป็นตาราง (Table) กึ่งกลางหน้าจอแบบกะทัดรัด ทำให้อ่านง่ายและไม่กินพื้นที่
3. **ป้ายกำกับตะกร้า (Basket Badge):** มีป้าย Badge บอกชื่อตะกร้าตั้งต้น (Source Basket) ที่ส่วนหัวของแต่ละรอบการแจกงาน ช่วยให้ตรวจสอบได้ทันทีว่าลูกค้ารอบนี้มาจากกลุ่มไหน

### 📥 การนำออกไฟล์ Excel (Best Practice Export)
การกดปุ่ม "ประวัติการแจกงาน" และคลิก "Export Excel" ระบบจะทำการดาวน์โหลดไฟล์ Excel ที่ประกอบด้วย 2 หน้า (Sheets):
1. **Summary Snapshot**: หน้าสรุปตารางการแจกงาน แสดงข้อมูลพนักงานที่เข้าเกณฑ์และได้รับรายชื่อ รวมถึงรายชื่อคนที่หลุดเกณฑ์ พร้อมเวลาโทร ณ วินาทีที่แจกงาน (รูปแบบพร้อมแคปส่งงาน)
2. **Customer Details**: ตารางรายชื่อลูกค้าแบบละเอียด (รหัส, ชื่อ, เบอร์โทร) แตกแจกแจงตามพนักงานที่ได้รับ เหมาะสำหรับนำไปกระทบยอดหรือตรวจสอบย้อนหลัง

> **📌 Data Retention Policy:** 
> ข้อมูลประวัติระดับรายละเอียดลูกค้า (`distribution_session_details`) จะขยายตัวขึ้นเรื่อยๆ ตามจำนวนการแจกงาน **แนะนำให้มีรันสคริปต์ Clean-up ลบข้อมูลรายละเอียดที่อายุเกิน 6-12 เดือนทิ้ง** เพื่อรักษาความเร็วของฐานข้อมูล (ส่วนข้อมูลภาพรวม Session จะยังคงอยู่)
