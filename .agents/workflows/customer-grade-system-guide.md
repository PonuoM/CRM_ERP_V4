# คู่มือระบบจัดการระดับลูกค้า (Dynamic Customer Grade System)

## 📌 ภาพรวม (Overview)
ระบบ **Dynamic Customer Grade System** คือระบบที่ออกแบบมาเพื่อให้แอดมินหรือผู้ดูแลระบบ (Admin) สามารถกำหนด "ระดับขั้นลูกค้า (Grade)", "ยอดสั่งซื้อสะสมขั้นต่ำ (Threshold)" และ "เงื่อนไขช่วงเวลาการคำนวณ (Calculation Mode & Time Range)" ได้อย่างอิสระตามแต่ละบริษัท (Multi-tenant) โดยไม่ถูกจำกัดอยู่กับเกรดพื้นฐานเดิม (D, C, B, A, A+) อีกต่อไป ผู้ใช้งานสามารถเพิ่มเกรดพิเศษ เช่น `VIP`, `S+` พร้อมกำหนดสีป้ายกำกับ (Badge Color) ได้ทันทีจากหน้าจอ UI รวมไปถึงตั้งค่าให้ระบบคิดยอดขายเฉพาะในช่วงเวลาที่กำหนดได้

---

## 🏗 โครงสร้างฐานข้อมูล (Database Schema)

### 1. ตาราง `customer_grades_config` (ตารางใหม่)
ตารางนี้ใช้สำหรับเก็บการตั้งค่าระดับเกรดของแต่ละบริษัท

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INT (PK) | รหัส Primary Key |
| `company_id` | INT | รหัสบริษัท (Multi-tenant) |
| `grade_name` | VARCHAR(50) | ชื่อเกรด เช่น `A+`, `S+`, `VIP` |
| `min_order_amount` | DECIMAL(10,2) | ยอดสั่งซื้อสะสมขั้นต่ำที่จะได้เกรดนี้ |
| `color_theme` | VARCHAR(50) | คลาส Tailwind สำหรับแสดงผลสี เช่น `bg-purple-100 text-purple-800` |
| `created_at` / `updated_at` | TIMESTAMP | เวลาที่สร้างและอัปเดต |

*(มี `UNIQUE KEY` คู่กันระหว่าง `company_id` และ `grade_name` เพื่อป้องกันเกรดชื่อซ้ำซ้อนในบริษัทเดียวกัน)*

### 1.5 ตาราง `customer_grades_settings` (ตารางใหม่)
ใช้เก็บเงื่อนไขการคำนวณยอดขายของแต่ละบริษัท

| Column | Type | Description |
| :--- | :--- | :--- |
| `company_id` | INT (PK) | รหัสบริษัท (Multi-tenant) |
| `calc_mode` | VARCHAR(20) | `all` (คิดทุกออเดอร์), `order_date`, `delivery_date` |
| `time_range_type` | VARCHAR(20) | `fixed` (ระบุวันแน่นอน), `relative` (นับย้อนหลัง x วัน) |
| `fixed_start_date` | DATE | วันที่เริ่มต้น (สำหรับ fixed) |
| `fixed_end_date` | DATE | วันที่สิ้นสุด (สำหรับ fixed) |
| `relative_days` | INT | จำนวนวันนับย้อนหลัง (สำหรับ relative) |

### 2. ตาราง `customers` (อัปเดตโครงสร้าง)
- **`grade`**: ปรับเปลี่ยนจาก `ENUM('D','C','B','A','A+')` เป็น **`VARCHAR(50)`** เพื่อรองรับชื่อเกรดใดๆ ก็ได้ที่ผู้ใช้งานกำหนดขึ้นมา

## ⚙️ โครงสร้างการทำงานของ Backend

Backend มีการอัปเดตให้รองรับแบบ Dynamic โดยมีส่วนสำคัญดังนี้:

1. **`api/Customers/get_grades_config.php`** (GET)
   - ดึงข้อมูลการตั้งค่าเกรดและ Settings (จาก `customer_grades_settings`) ตาม `company_id` ของ User ที่กำลังล็อกอิน
   - หากไม่เคยมีการตั้งค่า จะส่งค่า Default (D, C, B, A, A+) กลับไป

2. **`api/Customers/save_grades_config.php`** (POST)
   - บันทึกการตั้งค่าเกรดใหม่ และเงื่อนไขการคำนวณ (Settings) 
   - ทำงานภายใต้ Transaction โดยจะ `DELETE` ค่าเก่าของ `company_id` นั้นออกทั้งหมด แล้ว `INSERT` ข้อมูลเกรดและ `ON DUPLICATE KEY UPDATE` ข้อมูล Settings เข้าไปใหม่
   - ทำการเรียกใช้สคริปต์ Background (`recalculate_customer_stats_safe`) เพื่ออัปเดตเกรดลูกค้าทั้งหมดในบริษัทนั้นใหม่ทันทีเมื่อมีการเปลี่ยนเกณฑ์

3. **`api/Services/CustomerStatsHelper.php`**
   - มีฟังก์ชันย่อย `calculate_grade_purchases_amount($pdo, $customer_id, $company_id, $actual_total)` ที่ใช้ประมวลผลยอดสั่งซื้อสะสมตาม `calc_mode` และ Time Range ของบริษัท
   - อัปเดตฟังก์ชัน `calculate_customer_grade($pdo, $gradeTotal, $company_id)`
   - ดึงข้อมูลเกณฑ์จากตาราง `customer_grades_config` (พร้อมระบบ Cache ระดับตัวแปร Static ใน Request นั้นๆ เพื่อลดภาระ DB)
   - เช็คยอดขายสะสม (`$gradeTotal`) ไล่จากเกรดที่มียอดขั้นต่ำ "สูงสุด" ไปหา "ต่ำสุด" หากพบว่าผ่านเกณฑ์จะคืนค่าชื่อเกรดนั้นทันที

## 🖥 โครงสร้างการทำงานของ Frontend

1. **`types.ts`**
   - เปลี่ยนประเภท `CustomerGrade` จาก Enum ตายตัว ให้กลายเป็น `string` เพื่อป้องกัน Type Error เมื่อนำไปใช้กับเกรดชื่อใหม่ๆ
   - เพิ่ม Interface `CustomerGradeConfig` และ `CustomerGradeSettings`

2. **หน้าจอจัดการเกรด (`CustomerGradeManager.tsx`)**
   - เพิ่มส่วนตั้งค่า "เงื่อนไขการคำนวณยอดขาย" (โหมดคำนวณ และ รูปแบบเวลา)
   - เป็น UI ให้ผู้ใช้สามารถ กรอกชื่อเกรด, ยอดสะสมขั้นต่ำ และเลือกสี 
   - ระบบจะช่วย **Sort ยอดสั่งซื้อจากมากไปน้อย (Descending)** ให้อัตโนมัติเฉพาะ **ตอนที่กดปุ่มบันทึก (On Save)** เพื่อป้องกันปัญหาช่องกรอกข้อมูลกระโดดขณะพิมพ์
   - **(Best Practice UX):** เปลี่ยนการใช้ `window.confirm` ไปใช้ `ConfirmModal` เพื่อให้การแจ้งเตือนสวยงามและไม่ Block Main Thread
   - วางอยู่ใน Component ย่อยและถูกเรียกใช้ผ่านแท็บ (Tabs) ในหน้า `ManageCustomersPage.tsx`

3. **การแสดงผล (Customer Table & Dashboard)**
   - ในทุกๆ หน้าที่มีการโชว์ลูกค้า API ฝั่ง Backend จะทำการ Mapping ชื่อเกรดที่คำนวณมาให้อย่างถูกต้องแล้ว 
   - Frontend จะแสดงผลตามชื่อและดึงค่าสีมาประยุกต์ใช้ ทำให้ UI เปลี่ยนแปลงตาม Config โดยไม่ต้องแก้โค้ด

## 🔄 ตัวอย่าง Workflow การทำงานจริง (Hybrid Approach)

เพื่อประสิทธิภาพสูงสุดของระบบเซิร์ฟเวอร์ เราใช้สถาปัตยกรรมแบบ **Hybrid**:
1. **Real-time (เมื่อมีออเดอร์ใหม่):** ระบบจะอัปเดตเกรด "เฉพาะลูกค้ารายนั้น" ทันที เพื่อให้เซลส์เห็นเกรดล่าสุด
2. **Cron Job / Manual (เมื่อไม่มีออเดอร์):** เนื่องจากลูกค้าที่ไม่ได้ซื้อของนานๆ เกรดควรจะตกลง แต่ถ้าเขาไม่มาซื้อเลย ระบบ Real-time ก็จะไม่ถูกเรียก ดังนั้นแอดมินต้องกดปุ่ม "ประมวลผลใหม่" หรือตั้งเวลาให้เซิร์ฟเวอร์รันกวาดล้างข้อมูลทุกเที่ยงคืน

### ลำดับเหตุการณ์:
1. **แอดมินแก้ไขการตั้งค่า** 
   - สร้างเกรด `Super VIP` ขั้นต่ำ `150,000` บาท
   - ตั้งค่า Time Range แบบนับย้อนหลัง 30 วัน
   - กดปุ่ม "ประมวลผลเกรดลูกค้าทั้งหมด" (เพื่อให้มีผลทันทีกับลูกค้าทุกคนในบริษัท)

2. **ลูกค้ามีออเดอร์ใหม่สำเร็จ (Delivered)**
   - เมื่อเซลส์ปิดการขาย ระบบจะรวมยอดเฉพาะในช่วงเวลา 30 วันย้อนหลัง
   - หากยอดถึง `150,000` บาท ลูกค้าได้เลื่อนขั้นเป็น `Super VIP` ทันที

3. **ลูกค้าหายไปไม่มาสั่งซื้อ**
   - ผ่านไป 35 วัน ยอดเดิมหมดอายุ แต่ระบบยังไม่รู้เพราะไม่มีออเดอร์กระตุ้น
   - เมื่อถึงเวลาเที่ยงคืน Cron Job ดึงรายชื่อมาตรวจ พบว่ายอด 30 วันย้อนหลังเป็น 0 
   - ระบบจึงปรับลดเกรดลงมาเหลือต่ำสุด (เช่น D) โดยอัตโนมัติ

## 💡 สิ่งที่ต้องระวัง (Caveats & Troubleshooting)

- **Strict Mode ENUM:** ระวังหากเซิร์ฟเวอร์ใดใช้ฐานข้อมูลที่ยังไม่ได้รัน `ALTER TABLE customers MODIFY COLUMN grade VARCHAR(50);` เมื่อ Insert เกรดแปลกๆ ระบบจะ Error (ในไฟล์ Migration 009 ได้ระบุการ Alter ไว้แล้ว)
- **การเรียงลำดับ:** ถึงแม้ Frontend จะบังคับเรียงยอดสั่งซื้อขั้นต่ำจากมากไปน้อยแล้ว แต่ในระดับโค้ด Backend ก็จะมีการสั่ง `ORDER BY min_order_amount DESC` ซ้อนอีกที เพื่อการันตีว่าการประเมินผลยอดขั้นต่ำจะตรวจจากเกรดที่สูงที่สุดก่อนเสมอ
- **Cache / Reload:** การเปลี่ยนระดับลูกค้าจะไม่แสดงผลบน UI ของพนักงานแบบ Real-time (Websocket) หากลูกค้าที่เปิดค้างไว้มีการปรับเปลี่ยนเกรด พนักงานจำเป็นต้องกดรีเฟรชหรือโหลดข้อมูลชุดใหม่
