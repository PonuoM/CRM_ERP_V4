# เอกสารคู่มือระบบจัดการและรายงานการโทร (Call Management System)

เอกสารนี้อธิบายการทำงานของหน้า Frontend (React/TSX) ที่เกี่ยวข้องกับระบบการติดตามการโทรภายใน CRM/ERP โดยแบ่งเป็น 6 ไฟล์หลัก ซึ่งสามารถจัดกลุ่มได้ตาม **แหล่งที่มาของข้อมูล (Data Source)** ออกเป็น 2 ประเภทหลัก ได้แก่ 
1. ระบบดึงข้อมูลเรียลไทม์แคช/API จาก PBX (หน้าปกติ)
2. ระบบดึงข้อมูลจากการนำเข้าล็อกผ่าน CSV (หน้า V2)

---

## 1. กลุ่มหน้าเว็บระบบปกติ (อ้างอิงข้อมูลผ่าน API / Onecall PBX)

ระบบในหน้ากลุ่มนี้เดิมถูกสร้างขึ้นเพื่อต่อ API หรือดึงสรุปข้อมูลที่ผูกกับ Provider ผ่าน PBX โดยตรง หรือเรียกผ่าน Base API ฝั่งเดฟเวอร์ชันแรก

### 1.1 `CallHistoryPage.tsx` (ประวัติการโทรและฟังเสียงสนทนา)
- **จุดประสงค์:** ซัพพอร์ตการไล่ดูประวัติการโทรรายเบอร์ และสามารถกดปุ่ม Play เพื่อ "ฟังไฟล์เสียงบันทึกสนทนา (Voice Recording)" บนเบราเซอร์ได้ทันที
- **ลักษณะการทำงานและ Logic:**
  - สร้าง POST request ยืนยันตัวตน (Authentication) เพื่อเอา Token แบบ Basic Auth จาก Onecall PBX
  - ยิง API ขอข้อมูลประวัติการโทรตรงจาก Endpoint `/onecall/orktrack/rest/recordings` โดยระบบจะทำการกรอง (Filter) สิทธิ์เบอร์โทรตามลำดับชั้นผู้ใช้ (System Admin ดูได้ทั้งหมด, Supervisor ดูได้เฉพาะลูกทีม, Telesale ดูได้แค่เบอร์ตัวเอง)
  - มี Audio Player ภายในตัวคอยดึง Audio Stream ด้วย Authorization Header มาเล่นสด ๆ โดยไม่ต้องดาวน์โหลด

### 1.2 `CallDetailsPage.tsx` (หน้ารายละเอียดการโทรส่วนบุคคล - ดั้งเดิม)
- **จุดประสงค์:** แสดงตารางสรุปผลการปฏิบัติงานด้านการโทรรายบุคคลแบบพื้นฐาน
- **ลักษณะการทำงานและ Logic:**
  - ควบคุมการค้นหาผ่านเดือน-ปี และเชื่อมต่อไปที่ `Onecall_DB/get_call_overview.php`
  - ข้อมูลในหน้าแสดงผลมีข้อจำกัด เช่น แสดงรายการ โทรทั้งหมด, สายที่ได้คุย, เวลาทำงาน (วัน), และ เวลาโทรเฉลี่ยต่อวัน

### 1.3 `CallsDashboard.tsx` (หน้าภาพรวม Dashboard - ดั้งเดิม)
- **จุดประสงค์:** เป็นกระดานบริหารข้อมูลสรุปสถิตการโทรในเชิงกราฟ (Line Chart, Pie Chart) ประจำเดือน
- **ลักษณะการทำงานและ Logic:**
  - ดึงข้อมูลด้วย API อย่าง `get_dashboard_stats.php`, `get_daily_calls.php`
  - มีตรรกะสำหรับการดึง Onecall credentials คล้ายกับ CallHistoryPage เพื่อช่วยดึงค่าตัวเลขมาแสดงผล

## 2. กลุ่มหน้าเว็บ V2 (อ้างอิงข้อมูลจากฐานข้อมูลของระบบผ่านการ Import CSV)

ไฟล์กลุ่มที่มีคำว่า `V2` (รวมถึงระบบหน้า Import) ถูกสร้างขึ้นเพื่อ **แก้ปัญหาและข้อจำกัดของการดึง API** โดยเปลี่ยนให้เป็นการผู้ใช้ (Admin) นำเข้าข้อมูลประวัติการโทรแบบละเอียด(CSV จากระบบ PBX) แล้วให้หน้าต่างๆ อาศัยฐานข้อมูลกลางนี้ในการสร้าง Report ซึ่งให้มิติข้อมูลที่ลึกซึ้งและ Performance ดีขึ้น

### 2.1 `CallImportPage.tsx` (หน้าต่างนำเข้าข้อมูลรหัสล็อก)
- **จุดประสงค์:** เป็นศูนย์กลาง (เครื่องมือแอดมิน) สำหรับนำไฟล์ CSV จาก PBX มากองรวมและบันทึกใส่ระบบ Database (Table: `onecall_batches`, `onecall_logs`)
- **ลักษณะการทำงานและ Logic:**
  - มีฟีเจอร์ประเมินไฟล์ (Preview Data) แสดงตัวอย่างแถวก่อนทำการอัพโหลด
  - **Check Duplicates:** ตรวจสอบข้อมูลซ้ำซ้อนในตารางก่อน Import รวมถึงเช็คเบอร์พนักงานที่ผูกกับ User ในตาราง `users` เพื่อป้องกัน Agent จับคู่ชนกัน
  - ทำการ Mapping หรือจับคู่ Field เบอร์โทรของพนักงาน (`agent_phone`) ให้เข้ากับรหัสประจำตัวของ Telesale อัตโนมัติ เพื่อกระจายสู่ Report ปลายทาง

### 2.2 `CallDetailsPageV2.tsx` (รายละเอียดการโทร V2)
- **จุดประสงค์:** แสดงผลงานระดับบุคคลในแบบละเอียด (Granular Analytics) แทนที่เวอร์ชันเก่า
- **ลักษณะการทำงานและ Logic:**
  - ชี้ API ไปที่ `Onecall_DB/get_call_overview_v2.php`
  - **ความแตกต่างจาก V1:** โครงสร้างตารางเพิ่มมิติข้อมูลที่มาจากการอ่าน CSV Log แล้ว อาทิ จำนวนสายที่ **ไม่ได้รับ (Missed Calls)**, **สายโทรเข้า (Inbound)** / **สายโทรออก (Outbound)** รวมถึงการคำนวณอัตราส่วนเปอร์เซ็นต์อย่าง **Answer Rate (%)** และ Outbound Rate (%) ได้อย่างแม่นยำ 

### 2.3 `CallsDashboardV2.tsx` (ภาพรวม Dashboard V2)
- **จุดประสงค์:** นำเสนอภาพรวมการทำงานในรูปแบบกราฟ โดยอาศัยข้อมูลเชิงลึกจากการจัดการ CSV
- **ลักษณะการทำงานและ Logic:**
  - ชี้ API กลุ่ม V2 (`get_dashboard_stats_v2.php`, `get_talk_summary_v2.php` ฯลฯ)
  - **ความแตกต่างจาก V1:** กราฟสามารถแสดงหมวดหมู่ที่ละเอียดขึ้น เช่น โดนัทชาร์ตแยก "รับสาย/ไม่ได้รับสาย" (เทียบจากยอดเวลาสาย > 40 วินาทีคือจับว่าได้คุย), และโดนัทแยกตามประเภททิศทางการโทร "โทรเข้า/ออก/ต่อสายภายใน" 
  - มี Widget ตารางสรุปรายบุคคลห้อยแนบมาด้านล่าง (Leaderboard) ให้เทียบเคียงสถิติกันโดยตรง

## 3. ระบบย่อยส่วนกลาง (Global Components)

เพื่อเสริมการทำงานของระบบให้ราบรื่นในทุกหน้า ได้มีการนำ Component ส่วนกลางมาประยุกต์ใช้กับทุกหน้าของระบบ Call

### 3.1 `OnecallLoginSidebar.tsx` (แถบตั้งค่า Onecall และปุ่มลอย)
- **จุดประสงค์:** จัดการเรื่องการยืนยันตัวตน (Authentication) เพื่อเตรียม credentials สำหรับเข้าใช้บริการฝั่ง PBX 
- **ลักษณะการทำงานและ Logic:**
  - รูปแบบ UI เป็น Floating Button (ปุ่มฟันเฟืองสีน้ำเงินมุมล่างขวา) และเมื่อกดจะแสดงแถบ Sidebar เลื่อนขึ้นมาจากด้านขวา (Off-canvas sidebar)
  - ผูกทำงานร่วมกับตารางในฐานข้อมูลผ่าน `env_manager.php` เพื่อตรวจเช็คว่าบริษัทหรือ User นี้มีรหัสผ่านสำหรับเข้าถึง Onecall หรือยัง
  - Component นี้ถูกนำไป **ฝังลงในทุกหน้าของระบบหน้า Call (ทั้งหน้าดั้งเดิมและหน้า V2)** เพื่อให้ผู้มีสิทธิ์ใช้งานแอดมินสามารถเปิดดูสถานะและแก้ไขการเชื่อมต่อ PBX ได้ตลอดเวลา ไม่ว่าจะอยู่ลึกในหน้าไหนก็ตาม

## สรุปข้อเปรียบเทียบ

| หมวดหมู่ฟีเจอร์ | ระบบหน้าปกติ (Non-V2) | ระบบหน้า V2 |
|---|---|---|
| **กระบวนการทางข้อมูล** | ทริกเกอร์ API เข้าเซอร์วิส PBX แบบ On-demand / Cached DB | อ่านข้อมูลจากฐานข้อมูลของระบบ 100% หลังผ่านการอัพโหลด |
| **ตัวเชื่อม (Feeder)** | `CallHistoryPage` หรือ Automation จากแฝงเบื้องหลัง | `CallImportPage` (หน้าต่าง Import CSV โดยฝีมือแอดมิน) |
| **ความละเอียดข้อมูล** | พอสังเขป (เวลาโทรรวม, ยอดเข้าถึง) | ละเอียดสูง (แยกสายเข้า/ออก, ยอดไม่ได้รับสาย, % อัตรารับสาย) |
| **ความเสถียร** | ขึ้นอยู่กับสถานะของ PBX API Server ในขณะนั้น | เสถียรและเร็ว เพราะข้อมูลนอนอยู่ใน DB ระบบ ERP เรียบร้อยแล้ว |
